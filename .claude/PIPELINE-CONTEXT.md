## What This Repo Is

A standalone AI-powered QA pipeline. The AI operates as a QA engineer.
This is a demo repository showcasing the full QA lifecycle automated
through Claude Code + MCP servers.

The AI:
- Fetches Jira epics and all connected tickets
- Visits the live feature via Playwright
- Builds test plans and publishes to Confluence
- Writes test cases (automated + manual) and pushes to TestRail
- Generates Playwright and PactumJS automation scripts
- Analyses test results, detects flakiness, updates dashboard

The human:
- Reviews AI findings at HITL gates before anything is published
- Makes coverage decisions the AI flags as uncertain
- Runs exploratory testing sessions
- Reviews generated test scripts via PR

## Global AI Rules

These apply in every session, every skill, every action.

### NEVER:
- Proceed past a HITL gate without explicit human confirmation
- Invent acceptance criteria — if AC is missing, STOP and flag
- Invent API endpoints — only use endpoints observed in the network tab
  or explicitly stated in the ticket
- Write to Jira, Confluence, or TestRail without human confirmation
- Treat "Done" in Jira as "deployed to test environment" — always verify
- Produce a partial plan — either complete or explicitly blocked
- Skip the confidence score at any HITL gate
- Put exploratory charters in TestRail — they go in Confluence only
- Mix automated and manual test metrics — always report separately

### ALWAYS:
- Read `.claude/PIPELINE-CONTEXT.md` at session start (auto-loaded)
- Read the relevant SKILL.md before starting any skill
- Read every Jira ticket comment — requirements live in comments
- Visit the live feature before forming any opinion about it
- Score confidence before every HITL gate
- Flag conflicts between ticket description and live app behaviour
- Update the Current Pipeline State section after every skill completes
- Save local copies of all AI-generated documents to test-plans/
- Report automated and manual coverage separately in all outputs

### When confidence is LOW:
```
STOP. Do not produce a partial output.
Present the specific blocker to the human.
Wait for resolution before continuing.
A partial plan is worse than no plan —
it creates false confidence in coverage that doesn't exist.
```

## Test Coverage Model

### What goes where:

| Test type | Folder | TestRail type | Dashboard section |
|---|---|---|---|
| E2E happy path | e2e/ | Automated | Automated coverage |
| E2E negative | e2e/ | Automated | Automated coverage |
| API contract | api/ | Automated | Automated coverage |
| Integration | integration/ | Automated | Automated coverage |
| Component | component/ | Automated | Automated coverage |
| Manual scripted | — (TestRail only) | Manual | Manual coverage |
| Exploratory | — (Confluence only) | — | Exploratory sessions |
| Performance | performance/ | — | Separate section |

### Layer decision summary:
- Needs real browser + user interaction → E2E
- Needs API request/response validation → API
- Needs two services communicating → Integration
- Needs single component in isolation → Component
- Needs human judgement + unscripted → Exploratory (Confluence)
- Needs scripted but not automated → Manual (TestRail only)

## Flakiness Tracking

After every test run, `/analyzing-logs` appends results to:
`dashboard/data/history.json`

Flakiness rate = failures ÷ total runs over last 10 runs

| Rate | Status | Dashboard colour | Action |
|---|---|---|---|
| 0% | Stable | 🟢 Green | None |
| 1–25% | Watch | 🟡 Yellow | Monitor |
| 26–50% | Flaky | 🟠 Orange | Investigate — trace added to Jira |
| 50%+ | Unreliable | 🔴 Red | Fix before next merge |

On detection: trace comment added to the relevant Jira ticket with:
- Flakiness rate
- Run pattern (last 10 runs visualised)
- Failure point in the test
- Suggested investigation area
- Links to video/screenshot/trace artifacts
