# AI QA Pipeline — Concept & Context

## What This Is

This is a standalone AI-powered QA pipeline built as a demo repository.
It showcases how a Quality Engineer can use Claude + MCP servers to automate
the full QA lifecycle — from picking up a Jira epic through to publishing
test results to a live dashboard.

The AI acts as a QA engineer. You act as the QA lead reviewing its work.

---

## The Problem This Solves

Traditional QA bottlenecks:

```
Ticket arrives in QA column
        │
        ▼ (manual — hours)
QA reads ticket, visits app, figures out what to test
        │
        ▼ (manual — hours)
QA writes test plan, gets it reviewed
        │
        ▼ (manual — hours)
QA writes test cases, uploads to TestRail
        │
        ▼ (manual — hours)
QA writes automation scripts
        │
        ▼ (manual — post-run)
QA analyses failures, updates tickets, publishes results
```

Every handoff is manual. Every update to Jira/Confluence/TestRail is manual.
Flaky tests go untracked. Coverage gaps are invisible until something ships broken.

---

## What This Pipeline Does Instead

```
You: "/receiving-tickets EPIC-42"
        │
        ▼ (AI — minutes)
AI fetches all epic tickets, visits live feature,
captures API calls, assesses readiness, scores confidence
        │
        ▼ HITL GATE — you review, confirm
        │
        ▼ (AI — seconds)
Publishes intake doc to Confluence.
Updates every Jira ticket with summary + Confluence link.
        │
        ▼ (AI — minutes)
Builds full test plan — scope, objectives, strategy,
layer decisions (E2E/API/Integration/Component),
entry/exit criteria, risks, timeline
        │
        ▼ HITL GATE — you review the plan
        │
        ▼ (AI — seconds)
Publishes test plan to Confluence.
Updates Jira tickets with plan summary.
        │
        ▼ (AI — minutes)
Writes detailed test cases for TestRail.
Flags uncertain cases for your review.
        │
        ▼ HITL GATE — you review flagged cases only
        │
        ▼ (AI — seconds)
Pushes cases to TestRail.
Updates Confluence + Jira with TestRail links.
        │
        ▼ (AI — minutes)
Generates automation test scripts.
Places in correct folder (e2e/api/integration/component).
        │
        ▼ (CI/CD — GitHub Actions)
Runs tests on every PR. Captures video/screenshot/trace on failure.
        │
        ▼ (AI — post-run)
Parses results. Calculates flakiness. Flags patterns.
Updates TestRail, Jira, Confluence.
Deploys dashboard to GitHub Pages.
```

## Human in the Loop (HITL)

The AI does the work. You make the judgement calls.

| Gate | What AI presents | What you decide |
|---|---|---|
| Gate 1 — After ticket intake | Screenshots, API surface, readiness per ticket, conflicts, confidence scores | Proceed / correct / park blocked tickets |
| Gate 2 — After test plan | Full plan — scope, strategy, layer decisions, risks | Approve / adjust coverage before anything publishes |
| Gate 3 — After test case design | Flags uncertain cases only | Approve flagged cases / let rest through |
| Gate 4 — Before TestRail push | Exact list of what will be pushed | Confirm — hard to undo |

Nothing is written to Jira, Confluence, or TestRail without your confirmation.

## Skill Files — Design Principles

### Why skill files instead of prompts?

A prompt tells the AI what to do once.
A skill file tells the AI how to think about a whole category of work —
with rules, examples, guardrails, and checkpoints built in.

The difference:

```
Prompt: "Write a test plan for this ticket"
→ AI produces something. Quality unpredictable. No guardrails.
→ If AC is missing, AI invents requirements.
→ No HITL gate. No Confluence publish. No Jira update.

Skill file: /planning-tests
→ AI reads the intake document first
→ AI checks readiness before planning
→ AI applies coverage rules per test layer
→ AI scores its own confidence
→ AI stops if AC is missing — does not invent
→ AI presents plan at HITL gate before publishing
→ AI publishes to Confluence + updates Jira after confirmation
```

### Skill file structure

Every skill follows the same pattern:

```
SKILL.md (under 500 lines)
├── Frontmatter (name, description, tools, disable-model-invocation)
├── Purpose (what this skill does and does not do)
├── Global guardrails (what AI must/must not do)
├── Steps (numbered, sequential, explicit)
├── HITL gate (what AI presents, what human decides)
├── Publish step (what AI writes and where)
└── Progress checklist (tracks completion)

references/ (detailed rules that would make SKILL.md too long)
├── domain-specific rules
├── positive/negative examples
├── templates
└── checklists
```

### Positive/negative examples — why they matter

The most effective guardrail against AI hallucination is showing it
exactly what good and bad looks like, in the specific domain it is working in.

Generic rule: "Write testable acceptance criteria"
With example:
```
❌ BAD (AI will generate meaningless tests against this):
"The cart should work correctly"

✅ GOOD (AI can generate precise, verifiable tests against this):
"When a logged-in user adds an item to cart, the cart icon badge
increments by 1 and GET /cart returns the item in the items array
with correct productId, quantity, and price."
```

Every reference file in this pipeline contains both examples.
This is the primary mechanism for preventing hallucination.

## Dashboard

Live at: `https://<username>.github.io/<repo>/`

Run locally: `npm run dashboard`

Shows:
- Epic coverage % per epic
- Pass / Fail / Blocked per epic
- Coverage by layer (E2E / API / Integration / Component)
- Test run history over time
- Flakiness rate per test case with run history sparkline
- Failed test artifacts — video, screenshot, trace viewer link

Auto-updates on every GitHub Actions test run.
Confluence feature pages link directly to the dashboard.

---

## Flakiness Tracking

A test is flaky when it produces non-deterministic results across runs
with no code change.

The pipeline tracks flakiness automatically:
- Run history stored per test case in `dashboard/data/history.json`
- Flakiness rate = failures ÷ total runs over last 10 runs
- Thresholds: 0% stable / 1-25% watch / 26-50% flaky / 50%+ unreliable
- On flakiness detection: trace comment added to Jira ticket
- Dashboard shows flakiness column with sparkline of last 10 runs
- Failed test artifacts (video/screenshot/trace) linked per run

## Getting Started

```bash
# Install dependencies
npm install

# Install Claude Code skills
make install-skills

# Run the pipeline on a Jira epic
# (in Claude Code)
/receiving-tickets EPIC-42

# Run tests locally
npm run test:e2e
npm run test:api
npm run test:integration
npm run test:component

# View dashboard locally
npm run dashboard
```
